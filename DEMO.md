This document describes how to run the avalanche method in the analysis CLI across multiple r-candidates for a single message, evaluated over multiple `x` exponents up to a chosen threshold.

**Avalanche Multi-x Experiment (analysis)**

**What the experiment does**
- Selects a single plaintext message.
- Raises its ciphertext to a sequence of odd exponents `x` (one per “instance”).
- For each `(r, x)` pair, derives a candidate decryption via `(e*x)^{-1} mod φ(r)` and the HBC transform.
- Builds the avalanche tree from unique decryptions, then runs beam search.
- Records a beam match percentage against the true message and logs r‑candidate accuracy for comparison.

**Key knobs (config + CLI)**
- `analysis_batch_messages` (or `--batch-size`): number of `x` values to use per batch. This is the **threshold**.
- `analysis_batch_candidates`: number of r-candidates per batch.
- `analysis_batch_batches` (or `--batches`): number of batches to run.
- `same_r_batch`: set to `false` to use multiple r-candidates per batch.
- `ciphertext_modify`: set to `true` to apply `c^x mod N`.
- `message.is_random`: set to `false` and provide `message.fixed_message` for a single-message experiment.
- `bits`: analysis prime size; `--bits` overrides this at runtime.
- `bits_decrypt`: set via `--bits-decrypt` to control bit width (prevents leading‑zero truncation).
- `use_hamming_distance`: optional ordering for avalanche candidates (`--use-hamming-distance`).

**Recommended config setup**
Edit `config/rsa_config_small_batch.json` (or your chosen config):
- `engine.analysis_batch_enable`: `true`
- `engine.analysis_batch_messages`: e.g. `50` (threshold: number of `x` values)
- `engine.analysis_batch_candidates`: e.g. `50` (number of r‑candidates)
- `engine.analysis_batch_batches`: e.g. `10`
- `engine.same_r_batch`: `false`
- `engine.ciphertext_modify`: `true`
- `engine.message.is_random`: `false`
- `engine.message.fixed_message`: `"HeloWrld12345678"` (or your own)

**Run the experiment (manual)**
```bash
cargo run --bin analysis -- \
  --bits 56 \
  --bits-decrypt 128 \
  --tests \
  --crypto-rng \
  --batches 10 \
  --batch-size 50 \
  --ciphertext-modify \
  -c config/rsa_config_small_batch.json
```

**Run the experiment (scripted)**
- `scripts/run_small_batch_beam.sh` defaults to `--bits 56` and `--bits-decrypt 128`, and you can override those with `ANALYSIS_BITS` and `ANALYSIS_BITS_DECRYPT`.
- It records a session JSON per run and prints the path for use in the viewer.
```bash
RUNS=1 scripts/run_small_batch_beam.sh
```

**Public-key encrypted-file demo**
- `scripts/run_small_public_key_demo.sh` calls `scripts/do_prep_pgp.sh`, defaults to `MODULUS_BITS=512`, generates armored OpenPGP messages under `config/pgp/<modulus-bits>/`, extracts the first PKESK ciphertext into a generated small-batch-derived config, and then runs `scripts/run_small_batch_beam.sh` against that encrypted fixed-file input.
- It also stores `prep_blob_1.pkcs1_v1_5_block.hex` and `prep_blob_1.pkcs1_v1_5_payload.hex` beside `prep_blob_1.asc`, then fails unless the final `Avalanche global majority vote` hex equals the stored payload hex.
```bash
RUNS=1 scripts/run_small_public_key_demo.sh
```

**Inspect the results**
- The analysis run writes `session.json` (or the path you pass to `--session-json`).
- Open the viewer and use the **Beam vs R** tab to compare beam match % to r‑candidate accuracy.
```bash
python3 scripts/session_viewer.py logs/session_<timestamp>_seed_<seed>.json
```

**How the threshold maps to `x`**
- For each batch, `x` is generated by `odd_ciphertext_exponent(e, instance_idx, ...)`.
- The number of `x` values used is `analysis_batch_messages` (or `--batch-size`).

**Suggestions to improve the experiment**
1. Increase **unique decryptions**:
   - Raise `analysis_batch_messages` and `analysis_batch_candidates`.
   - Increase `process_count` / `process_min_count` for r‑candidate generation.
2. Reduce duplicate `(e*x)^{-1}`:
   - Increase `analysis_batch_messages` first; duplicates are filtered per candidate.
3. Stabilize bit width:
   - Always set `--bits-decrypt` to the expected message width.
4. Compare ordering strategies:
   - Toggle `--use-hamming-distance` to see its effect on beam match.
5. Diversify r‑candidates:
   - Set `same_r_batch: false` and increase `analysis_batch_candidates`.
   - Vary `r_candidate_small_primes` and `r_candidate_small_prime_factors`.
