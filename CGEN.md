# CGEN

## Purpose

`cgen` is a standalone generator for precomputing compact combination indexes. It writes up to `K` combinations of `N` indices chosen from a universe of `Q` values into a CSV file. The output is designed to be small enough to persist on disk and fast enough to decode later inside analysis code.

The implementation lives in [src/bin/cgen.rs](/home/nlaroche/projects/rsa/rsa-statistical-advantage/src/bin/cgen.rs).

## Background

The longest-running modern `analysis` path is the batch scoring plus sampled-avalanche stage. After each batch has produced scored candidate decryptions, analysis repeatedly:

1. groups the scored outputs by `r` candidate,
2. selects many sampled combinations,
3. runs avalanche reduction for each sample,
4. converts the result into beam-search probabilities,
5. and keeps the best sample for reporting.

That loop is implemented in [src/methods.rs](/home/nlaroche/projects/rsa/rsa-statistical-advantage/src/methods.rs#L4998) and described in [ARCHITECTURE.md](/home/nlaroche/projects/rsa/rsa-statistical-advantage/ARCHITECTURE.md#L115). In practice, this stage can dominate runtime when `avalanche_combination_samples` is large, because analysis evaluates many sample layouts per batch.

`cgen` exists to support that workload by letting the project precompute sample membership offline instead of rebuilding combination layouts on demand. The compressed CSV format is meant to make those layouts cheap to store and cheap to stream back in.

## Current Status

As of the current codebase, `cgen` is available as a generator, but `analysis` does not yet read this CSV. The sampled-avalanche path still selects groups at runtime with `select_scored_inputs_for_mixed_r_candidates` in [src/methods.rs](/home/nlaroche/projects/rsa/rsa-statistical-advantage/src/methods.rs#L4927).

So the performance story is:

- current behavior: `analysis` samples combinations in memory during each batch,
- intended use: precompute reusable combination layouts with `cgen`, then load and decode them in the avalanche sampling stage.

## CLI

`cgen` accepts four arguments:

- `--universe-size` / `-q`: `Q`, the number of available values, corresponding to indices `0..Q-1`
- `--combination-size` / `-n`: `N`, the number of indices in each combination
- `--limit` / `-k`: `K`, the maximum number of combinations to write
- `--output` / `-o`: destination CSV path

Example:

```bash
cargo run --bin cgen -- \
  --universe-size 5 \
  --combination-size 3 \
  --limit 5 \
  --output data/cgen_output.csv
```

## CSV Schema

The CSV header is exactly:

```text
combination_index,compressed_indices_hex
```

Each subsequent row has two columns:

1. `combination_index`
2. `compressed_indices_hex`

### `combination_index`

- Type: unsigned decimal integer
- Meaning: zero-based lexicographic position of the combination among all `N`-of-`Q` combinations generated by `cgen`
- Ordering: rows are written in ascending `combination_index` order

For `Q = 5` and `N = 3`, the first few combinations are:

```text
0 -> [0,1,2]
1 -> [0,1,3]
2 -> [0,1,4]
3 -> [0,2,3]
4 -> [0,2,4]
...
```

### `compressed_indices_hex`

- Type: lowercase hexadecimal string
- Meaning: hex encoding of a binary payload
- Payload format: gap-encoded, unsigned LEB128 varints

The original combination indices are always sorted and strictly increasing.

## Binary Encoding

Given a combination:

```text
[i0, i1, i2, ..., iN-1]
```

`cgen` converts it into non-negative gap values:

```text
g0 = i0
g1 = i1 - i0 - 1
g2 = i2 - i1 - 1
...
```

Each `gk` is then encoded as an unsigned LEB128 integer, and the bytes are concatenated. The CSV stores that byte sequence as lowercase hex.

This is implemented in [src/bin/cgen.rs](/home/nlaroche/projects/rsa/rsa-statistical-advantage/src/bin/cgen.rs#L256).

## Decoding

To decode one row:

1. Parse `compressed_indices_hex` into raw bytes.
2. Read exactly `N` unsigned LEB128 integers.
3. Reconstruct indices:

```text
i0 = g0
ik = i(k-1) + gk + 1   for k >= 1
```

The decoder must already know `N`. The CSV does not store `Q`, `N`, or `K` inside each row.

That means the file must be interpreted together with the generation parameters used to create it.

## Guarantees

For the current implementation:

- CLI validation requires `Q >= 1`
- CLI validation requires `N >= 1`
- runtime validation requires `N <= Q`
- if `K == 0`, the CSV contains only the header row
- if `K` exceeds the total number of combinations, `cgen` writes all available combinations
- the file is overwritten if the output path already exists

## Example

For:

```text
Q = 5
N = 3
K = 5
```

the CSV begins as:

```csv
combination_index,compressed_indices_hex
0,000000
1,000001
2,000002
3,000100
4,000101
```

Those rows decode as:

- `000000` -> gaps `[0,0,0]` -> indices `[0,1,2]`
- `000001` -> gaps `[0,0,1]` -> indices `[0,1,3]`
- `000002` -> gaps `[0,0,2]` -> indices `[0,1,4]`
- `000100` -> gaps `[0,1,0]` -> indices `[0,2,3]`
- `000101` -> gaps `[0,1,1]` -> indices `[0,2,4]`

## Why This Helps Analysis

The sampled-avalanche stage in `analysis` currently spends time deciding which candidate groups belong to each sample before avalanche and beam search are run. A precomputed combination index lets the runtime skip repeated combination construction and instead:

1. read a row,
2. decode `N` compact gaps,
3. map those decoded indices onto the batch-local candidate pool,
4. run avalanche and beam search immediately.

That does not remove the cost of avalanche reduction or beam search themselves, which remain the dominant computation, but it can reduce per-sample overhead and makes sample selection deterministic and reusable across runs.
